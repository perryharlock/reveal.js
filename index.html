Branch structures and workflow

A few ideas...

Currently...

We create a single branch for all stories in the sprint from trunk.

And then...

Once stories are complete in theory a code review will happen.

And then...

Once reviewed and OK'd changes are pushed to trunk and deployed to test.

So what's wrong with that?

  • We get to the point where 'ALL TEH THINGS' are being tested at once towards the end of the sprint.
  • This can lead to a last minute rush to make amends.
  • If one of the stories is incomplete we cannot deploy until it is finished.
  • It is best to code review small chunks of work rather than a whole sprints worth.

Ok, what do you propose Charlie Big Potatoes?

The following proposal is based mainly from how other Agile / Scrum environments I have worked in operate.

I've also talked to people in other organisations who work in a similar fashion to the way I am about to describe.

IT'S NOT GOSPEL!

Use Git for source control

This I supposes is a personal preference however I have found that Git is quicker than TFS for branching, committing, etc... (once you have the command line licked!)

Git

For example creating a new branch is as easy as typing: 

git checkout -b "new-branch-name"
from the branch you want to branch from.

GitHub

I'd also love to use GitHub. Mainly for code reviews (but I understand there may well be a technical issue here)

Creating a Pull Request which shows exactly what has changed for code reviewing is awesome!

Code also does not slip under the radar until the PR has been OK'd by a reviewer.

Branching

  • Creating a master branch for the sprint.
  • Create a single branch for each story.
  • As each story is completed it goes to code review.
  • Once code reviewed it can be pushed to the master sprint branch which can in turn be pushed to TEST.
  • The story branch is then deleted.
  • Repeat for each story

Hmmm....what are the benefits of this?

  • Code can be deployed to TEST at any time for testing and feedback not just 2 days before the end of the sprint.
  • If a story is not complete we can still deploy without it.
  • We could deploy to LIVE more often if we want to release an individual story early.

Guidelines

  • No editing on trunk / master except for urgent bug fixes.
  • Code does not get pushed to TEST until it has been code reviewed and OK'd.
  • Incomplete / Broken code does not get pushed to TEST.
  • Branch naming: 
    #JIRATICKETNUMBER-name-of-story
